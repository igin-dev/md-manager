# Принципы написания MD-файлов для Claude Code

## Главное правило

**В CLAUDE.md идёт только то, что агент НЕ МОЖЕТ вывести из кода сам.**

Это единственный критерий. Всё остальное — следствие из него.

Агент читает файлы проекта, видит структуру классов, паттерны логирования, команды в package.json или Makefile. Не нужно дублировать это в CLAUDE.md — это увеличивает контекст без пользы и создаёт риск рассинхронизации с реальным кодом.

## Что точно НЕ НУЖНО в CLAUDE.md

- Команды запуска если они очевидны (`python app.py`, `npm start`) — агент это выведет
- Формат логов если он виден в коде — агент его прочитает
- Стандартные паттерны языка (`try/except`, `async/await`) — агент их знает
- Структура классов и модулей — агент читает файлы напрямую
- Общие best practices не специфичные для проекта

## Что точно НУЖНО в CLAUDE.md

**Навигация** — что агент не найдёт без подсказки:
- Карта субпроектов с путями к их CLAUDE.md (в монорепо)
- Какие файлы читать перед изменением конкретного компонента
- Внешние зависимости не видные из кода (Task Scheduler, systemd, cron)

**Gotchas** — неочевидное поведение которое сломает работу:
- Нестандартное использование библиотек (asyncio.run() ломает event loop)
- Платформенные особенности (T-формат дат ломает SQLite-сортировку)
- Архитектурные решения принятые по неочевидным причинам

**Явные запреты** — только если их нарушение не очевидно:
- Запрет использовать библиотеку X когда есть более очевидный выбор
- Запрет паттерна который выглядит логично но несовместим с архитектурой

**Намеренные нестандартные решения**:
- "Комментарии на русском" — агент не угадает это из кода, если не видел достаточно файлов
- Нестандартный стиль именования принятый в команде
- MCP-серверы — агент не знает что они доступны без явного указания

**Команды запуска** — только если они нестандартны или важен контекст:
- Когда есть несколько способов запуска и важно выбрать правильный
- Когда что-то уже запущено автоматически и повторный запуск опасен
- Когда есть неочевидный порядок запуска компонентов

## Тест перед добавлением строки

Задай себе вопрос: "Агент прочитает 5 случайных файлов проекта — поймёт ли он это без CLAUDE.md?"

- Если **да** — не добавляй
- Если **нет** — добавляй, кратко и с причиной

## Размер

| Тип | Рекомендуемый размер |
|---|---|
| Корневой CLAUDE.md (монорепо) | 20–60 строк |
| CLAUDE.md субпроекта | 30–80 строк |
| CLAUDE.md инструмента | 50–150 строк |

Больше — только если проект реально сложный с множеством gotchas.
Если файл растёт — выноси в @ARCHITECTURE.md или @CONVENTIONS.md.

## Стиль

Императивный и конкретный. Агент следует инструкциям буквально.

| Не писать | Писать |
|---|---|
| Рекомендуется использовать TA-Lib | Используй только TA-Lib — не vbt built-in |
| Лучше проверять timezone | Все datetime в БД — naive UTC |
| Можно запустить через python app.py | *(не писать — очевидно из кода)* |
| Используй RotatingFileHandler | *(не писать если это уже в коде)* |

Без капслока кроме 1–2 акцентов. Без восклицательных знаков. Язык однородный.

## Минимальная структура CLAUDE.md

Нет обязательных секций. Включай только те что нужны:

```markdown
# Название — одна строка что это

## Навигация (если монорепо или сложная структура)
Карта субпроектов, что читать перед изменениями.

## Запуск (только если нестандартный или важен контекст)
Конкретные команды с указанием КОГДА какую использовать.

## Внешние компоненты (если есть)
Task Scheduler, cron, systemd — что уже запущено автоматически.

## MCP-серверы (если настроены)
Список с коротким описанием для чего каждый.

## Gotchas
Только неочевидное. С примером и причиной.

## Запреты
Только что выглядит логично но сломает проект.
```
