# Принципы написания MD-файлов для Claude Code

## Для кого пишется CLAUDE.md

CLAUDE.md читает AI-агент, не человек. Это принципиально меняет стиль:
- Агент следует инструкциям буквально
- Агент не "понимает контекст" если он не написан явно
- Агент не знает что очевидно из кода — только то что написано
- Агент тратит токены на каждую строку — каждая должна нести информацию

## Размер

| Тип файла | Рекомендуемый размер | Максимум |
|---|---|---|
| Корневой CLAUDE.md (монорепо) | 80–150 строк | 200 строк |
| CLAUDE.md субпроекта | 60–120 строк | 150 строк |
| CLAUDE.md инструмента | 100–200 строк | 300 строк |
| ARCHITECTURE.md | без ограничений | — |
| CONVENTIONS.md | без ограничений | — |

Если нужно больше — выносить детали в referenced файлы через `@filename`.
Claude Code подгружает referenced файлы лениво, только при обращении агента.

## Стиль

**Императивный:** агент следует инструкциям, не интерпретирует пожелания.

| Не писать | Писать |
|---|---|
| Рекомендуется использовать TA-Lib | Используй только TA-Lib для индикаторов |
| Лучше оборачивать в try/except | Оборачивай в try/except + logger.error везде |
| Желательно проверять timezone | Все datetime в БД — naive UTC. Конвертируй через ta_timezone.py |
| Можно запустить через python app.py | Запуск: `python app.py` |

**Конкретный:** не "используй лучшие практики", а "используй RotatingFileHandler(maxBytes=5MB)".

**Плотный:** каждая строка несёт новую информацию.
Убирать: вступления, повторения, слова-паразиты ("важно отметить", "следует учитывать").

**Без эмоций:** восклицательные знаки и капслок не улучшают следование инструкциям.
Максимум 1–2 слова в капслоке для критичных акцентов (ЗАПРЕЩЕНО, НИКОГДА).

**Однородный язык:** русский или английский. Технические термины на английском
внутри русского текста — нормально. Целые предложения вперемешку — нет.

## Что включать

**Включай:**
- Команды запуска — конкретные, с указанием когда нужны
- Gotchas — что неочевидно из кода, с примером и причиной
- Запреты — с объяснением почему запрещено и что делать вместо
- MCP-серверы — с примерами конкретных инструментов
- Ссылки на другие файлы через @

**Не включай:**
- То что очевидно из кода (структура классов, стандартные паттерны)
- То что есть в родительском CLAUDE.md (дублирование)
- Общие best practices без привязки к проекту
- Историю изменений (для этого есть git)

## Структура CLAUDE.md

```markdown
# Название — краткое описание (одна строка)

## Запуск

| Сценарий | Команда |
|---|---|
| ... | `...` |

## Стек

- **Ключевая библиотека**: ограничение или правило использования
- **Запрещено**: что нельзя использовать и почему

## Запреты

**Что запрещено** — причина. Альтернатива: что использовать вместо.

## Архитектурные gotchas

**Название gotcha**: описание проблемы и правильное решение.

## Ссылки

@ARCHITECTURE.md — читать перед изменением архитектуры
@CONVENTIONS.md — соглашения по коду
```

## Структура ARCHITECTURE.md

```markdown
# Architecture — Название проекта

## Обзор

2–3 предложения что это, как работает, какую задачу решает.

## Компоненты

ASCII-схема или таблица компонентов с ответственностью и файлами.

## Потоки данных

Описание основных потоков: A → B → C с пояснением каждого шага.

## Ключевые решения

Что сделано и почему именно так. Альтернативы которые не подошли.

## Ограничения

Известные ограничения и их причины.
```

## Структура CONVENTIONS.md

```markdown
# Conventions — Название проекта

## Код

Стиль, паттерны, что использовать / не использовать.

## Именование

Файлы, переменные, функции, классы.

## Обработка ошибок

Конкретный паттерн с примером кода.

## Логирование

Настройка, формат, уровни.

## Тестирование

Где тесты, как запускать, что покрывать.
```
