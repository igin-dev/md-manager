# Критерии оптимизации MD-файлов

## Главный вопрос

Для каждой строки файла: "Агент выведет это из кода сам?"
Если да — это кандидат на удаление, а не на сжатие.

## Что удалять (очевидно из кода)

- Стандартные команды запуска — `python app.py`, `npm start`, `flask run`
- Форматы логов если они настроены в коде
- Стандартные паттерны языка — try/except, async/await, структуры классов
- Общие best practices без привязки к специфике проекта
- Описание структуры кода — агент читает файлы напрямую

## Что оставлять всегда

- Автозапуск через Task Scheduler, cron, systemd — не видно из кода
- Gotchas с неочевидным поведением — с примером и причиной
- Запреты с объяснением почему нельзя то что выглядит логичным
- MCP-серверы — агент не знает о них без явного указания
- Навигационная карта субпроектов в монорепо

## Что выносить в referenced файлы

Только если контент реально нужен, но занимает много места:
- Детальные справочные данные (больше 20 строк) → @CONVENTIONS.md
- Архитектурные схемы и потоки данных → @ARCHITECTURE.md

Не выносить:
- Gotchas — должны быть под рукой, не в другом файле
- Критичные запреты — то же самое
- Навигацию по субпроектам

## Что сжимать (только после удаления лишнего)

Список команд в таблицу — если команды реально нужны и их больше двух.
Убрать вводные слова: "важно отметить", "следует учитывать", "необходимо помнить".
Объединить повторяющийся контекст в один заголовок секции.

## Что никогда не трогать

Gotcha с примером кода — пример там не случайно.
Объяснение причины запрета — без причины запрет теряет смысл.
Специфичные значения параметров (maxBytes=5MB, fees=0.001) — они критичны.
